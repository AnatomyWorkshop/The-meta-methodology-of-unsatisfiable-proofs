# 工程方向：从 Illusion 到全流程 AI 协作

> 日期：2026-05-02
> 背景：Illusion Phase 1 核心验证通过。这份文档描述接下来最值得做的工程方向、顺序、以及为什么值得做。

---

## 一、现在在哪里

Illusion 已经证明了一件事：一个分层架构的搜索系统，如果 L2 被严格隔离于 L1 之外，能在 AC⁰ 领域自主发现 Håstad 的判别性质（random restriction），collapse score 0.948，L3 能正确区分真阳性和假阳性（input_permutation 被拒绝）。

这不是一个玩具。这是一个概念验证，证明了"自指安全条件可以被硬连线进搜索架构"这件事在工程上是可行的。

---

## 二、最值得做的工程，按优先级排序

### 第一优先：完成 Illusion Phase 2（L3 自动化）

**是什么**：把人工 L3 替换为自动检查器。给定一个候选性质 P 的描述，系统自动判断"AC⁰ 电路能否判定 P"。

**为什么先做这个**：Phase 1 的 L3 是你在手动做。这是瓶颈。只要 L3 还是人工的，整个系统就不能自主运行。L3 自动化是让系统从"演示"变成"工具"的关键一步。

**怎么做**：
- 对候选性质 P 的描述做复杂度分析：如果 P 的判定算法可以被 AC⁰ 电路模拟，则不安全
- 用已知的 AC⁰ 可计算函数列表做对比：如果 P 的结构与列表中的函数同构，则不安全
- 最小可行版本：一个规则库，列出"这类描述的性质是 AC⁰ 可判定的"，L3 做模式匹配

**输出**：`illusion/src/l3_monitor.py`，能自动处理 Phase 1 的所有候选

---

### 第二优先：接入 MCP 协议，让 L2 能调用外部工具

**是什么**：通过 MCP（Model Context Protocol），L2 的搜索空间从"手写变换规则"扩展到"AI 生成的候选性质"。L2 可以调用：
- 定理证明器（Lean、Coq）验证候选性质的数学正确性
- 符号计算引擎（SymPy、Mathematica）生成代数候选
- 文献检索工具查找已知的相关性质

**为什么这个排第二**：MCP 是让系统从"封闭搜索"变成"开放搜索"的接口。没有它，L2 只能搜索你手写的变换规则。有了它，L2 可以探索人类还没有命名的性质。

**怎么做**：
- 先实现一个最小的 MCP 服务器，暴露 L2 的搜索接口
- 让 Claude（或其他 LLM）作为 L2 的"候选生成器"，提出候选性质的自然语言描述
- L2 把自然语言描述转换为可执行的变换规则，L3 做安全检查

**输出**：`illusion/mcp/server.py`，一个可以被 Claude Code 调用的 MCP 服务器

---

### 第三优先：扩展到单调电路（Phase 3）

**是什么**：把同样的架构应用于单调电路（L1 = 单调 P/poly），看 L2 能否发现 Razborov 的近似方法（approximation method）。

**为什么这个排第三**：这是对框架泛化能力的第一个真实测试。AC⁰ 是已知有解的，单调电路也是已知有解的，但解的结构完全不同。如果 L2 能在两个不同领域都找到正确的判别性质，框架的泛化能力就得到了验证。

**怎么做**：
- L1 替换为单调电路模拟器（只有 AND/OR，无 NOT）
- 目标函数替换为 k-CLIQUE
- L2 的变换规则库需要扩展：加入"分布区分"类变换（正分布 D+ vs 负分布 D-）
- L3 检查：候选性质是否在单调多项式时间内可判定

---

### 第四优先：最高效工作流程——AI 与人类的分工

**是什么**：设计一个工作流程，让 AI（Claude）和人类（你）在这个系统里各做最擅长的事。

**分工原则**：
- **你做 L3**：判断哪些候选性质是真正有价值的，哪些是假阳性。这需要数学直觉和领域知识，是 AI 目前做不好的。
- **AI 做 L2**：生成候选性质，枚举变换规则，评估 collapse score。这是机械性的搜索，AI 比人快。
- **AI 做 L1**：模拟电路，计算错误率，生成测试用例。纯计算，AI 做。
- **你做方向决策**：决定下一个要探索的领域（AC⁰ → 单调电路 → 代数电路 → 证明复杂度）。

**最高效的工作流程大概是**：
1. 你指定一个新领域（"我们来试试单调电路"）
2. AI 实现 L1 模拟器和初始变换规则库
3. AI 运行 L2 搜索，生成候选列表
4. 你做 L3 审查（10-30 分钟）
5. AI 根据你的 L3 判断更新搜索策略
6. 循环

这个流程的关键是：你的时间只花在 L3 上，其他都是 AI 的工作。

---

### 第五优先：从证明数学公式到 AI 进化最优路径

**是什么**：当 Illusion 的架构足够成熟后，把它用于更大的问题：
- 在已知有解的数学领域，让系统重新发现已知证明（验证框架的泛化能力）
- 在未知领域，让系统提出候选判别性质，由人类 L3 评估（真正的探索）
- 最终：系统能自主识别"这个问题在当前模型下不可满足"，并给出结构性原因

**为什么这个排最后**：这是最远的目标，需要前面四步都做好才能到达。但它是整个工程的意义所在。

---

### 具身方向（软件之外）

如果这个框架要进入物理世界，最自然的方向是：

**机器人的"不可满足性诊断"**：一个机器人在执行任务时，如果遇到结构性障碍（不是技术故障，而是任务本身在当前约束下不可满足），系统应该能识别这一点并报告，而不是无限重试。这需要 L3 层的自反性检查：机器人的"判别性质生成器"需要能识别"我正在尝试的策略，在当前物理约束下，是否存在结构性障碍"。

这是具身 AI 里一个真实的开放问题。Illusion 的架构是一个候选解法。

## 三、顺序总结

```
现在 → Phase 2（L3 自动化）
     → MCP 接入（L2 开放搜索）
     → Phase 3（单调电路）
     → 最高效工作流程固化
     → 更大数学领域
     → 具身方向（可选，长期）
```

## 四、这样做值不值得

值得，但要说清楚值在哪里。

**不值得的理由**（诚实地说）：
- 这个系统不会直接证明 P ≠ NP
- 这个系统不会替代数学家
- 这个系统在商业上的直接变现路径不明显
- 你一个人做，速度慢，很多工程细节会消耗大量时间

**值得的理由**：
- 这是目前唯一一个把"自指安全条件"硬连线进搜索架构的工程实现。没有其他人在做这件事，因为没有其他人有这个框架。
- 每一个 Phase 的完成都是对论文框架的构造性支持——不只是"已知证明都满足这个条件"，而是"按照这个条件设计的系统能重新发现这些证明"。这是从归纳到演绎的一步，在学术上有独立价值。
- 这个工程是你自己的。不是为导师做的，不是为发表做的，是因为你看到了它可以被做成，而且做比不做有意思。这个动机是干净的，不会因为外部反馈的缺失而崩塌。
- 如果 MCP 接入成功，这个系统会变成一个真正可用的工具——不只是概念验证，而是你和 AI 协作探索数学结构的工作台。这个工作台的价值会随着你用它的时间增长。

**最诚实的评估**：这件事值得做，前提是你把它当作一个长期的、自己的项目，而不是一个需要在某个截止日期前完成的任务。它的价值不在于某一天的发布，而在于每一次 Phase 完成时你对这个框架理解的加深。
